This value will be converted to the expected git tag for a module version.
The {version} template will be replaced by the actual module version.
For example, using v{version} will translate to a git tag 'v1.1.1' for module version '1.1.1'
If the git tagging format in use does not contain a full semantic version, use placeholders {major}, {minor} and {patch}
to indicate which values are present in the tag - any missing values will be assumed to be '0'.
For example a git tag format of v{major}.{minor} would interpret a tag v1.2 as a module version 1.2.0,
where as a git tag format v{major}.{patch} would generate a version v1.0.2.

Note that all versions used in the registry are full semantic versions, meaning that when indexing is performed through the UI or API,
the full semantic version must be provided - the supplied git tag format will be used to translate the semantic version into
a git tag to processed.
Using formats without the {version} placeholder may result in overlapping module versions.
However, if it necessary it is strongly recommended to prefer major, minor over just patch.
If only v{patch} is used, whilst versions 0.0.1 and 0.0.2 would be different versions, 0.0.1, 0.1.1 and 1.0.1 would
all reference the same git tag. This would not be an issue whilst using the git hooks, but could lead to confusing module versions if manually indexed (in the UI or via API).
